## 背景  

> Lance 接入 DuckDB 的意义，不只是多了一个文件格式，而是把“分析型 SQL”和“AI 检索型访问”放回同一张数据表里。

## 以前 DuckDB 只擅长顺序读

传统湖仓格式的核心假设是：数据主要被批量扫描、过滤、聚合。这个假设在 BI、报表、离线数仓里成立，所以 Parquet 这样的列式格式长期有效。

AI 数据把这个假设打破了。一个图文检索表里，既有标量字段，也有长文本、图片字节、音频、768 维向量、全文索引、向量索引和不断追加的派生特征。查询也不再只是 `WHERE` 和 `GROUP BY`，而是“找相似图片”“找语义接近的文本”“先按业务条件过滤，再做向量召回”。这类访问模式需要随机读、索引维护、版本演进和事务更新。

Lance 的关键价值就在这里。它把文件格式、表格式和轻量 catalog 放在同一套开放格式里，官方表格式文档明确支持 ACID transaction、schema evolution、time travel，并通过 MVCC 做增量更新；其 DuckDB 扩展则把读写 Lance、向量搜索、全文搜索和混合搜索暴露为 SQL 能力。[DuckDB 文档](https://duckdb.org/docs/current/core_extensions/lance.html)也已经把 `lance_vector_search`、`lance_fts`、`lance_hybrid_search` 作为扩展能力列出。

```mermaid
flowchart LR
  A["传统湖仓: 扫描/聚合"] --> B["AI 数据: 文本/向量/图片/元数据混合"]
  B --> C["访问模式变化: 顺序读 + 随机读 + 检索排序"]
  C --> D["格式要求变化: 表版本 + 索引 + SQL 内检索"]
  D --> E["DuckDB + Lance: 分析和召回同一入口"]
```

## Lance 补足 AI 数据访问层

这次组合最容易被误读成“Lance 比 DuckDB 快”。更准确的说法是：DuckDB 继续做嵌入式 SQL 计算引擎，Lance 承担更适合 AI 数据的存储、索引和随机访问层。

原文 LAION 子集实验很能说明边界：在冷启动下，Lance native 的向量索引和混合搜索明显快于 Parquet，也优于 DuckDB indexed；但在 warm run 中，DuckDB indexed 的向量索引也非常快，全文检索和精确向量搜索并非 Lance 全面碾压。这个结果不该被读成单点跑分胜负，而应读成架构分工：当数据只是临时分析，DuckDB 自身格式和扩展足够强；当同一份数据要长期保存、持续更新、管理向量/全文索引，并被检索系统反复访问，表格式能力开始变得重要。

这也是 Lance 的论文问题意识。2025 年的 Lance 论文把矛盾说得更底层：列式格式既要高吞吐扫描，又要高效随机访问，结构编码会直接影响 NVMe 上的实际性能。换句话说，AI workload 不是在湖仓之外多挂一个向量库那么简单，它正在改变底层存储格式的优化目标。

## 生产意义

AI 应用里最贵的经常不是某个查询慢 20 毫秒，而是同一份数据被复制到多个系统：对象存储放原始图片，Parquet 放元数据，向量库放 embedding，全文引擎放文本索引，业务库再存版本状态。每多一份副本，就多一层一致性、权限、回填和生命周期问题。

DuckDB + Lance 的方向，是让工程师在 SQL 工作流里同时完成分析、过滤、召回和表维护。DuckDB 的 Lance 扩展支持本地路径、对象存储、目录 namespace，也支持通过 REST namespace 接入远端 catalog；Lance 表格式则提供版本、schema 演进和事务语义。这意味着从个人实验、离线评估到远端生产 catalog，可以尽量保持同一个数据模型。

但边界也要清楚。Lance 不是要取代所有湖仓表格式，也不是所有分析负载都需要它。如果数据主要是结构化指标、宽表扫描和批量聚合，Parquet/Iceberg/Delta 仍然是成熟选择。Lance 更适合那些“检索就是主路径”的数据：RAG 语料、多模态素材库、embedding 特征库、视觉搜索、推荐候选集、模型评估样本池。

## 接下来该看什么

- DuckDB Lance 扩展是否继续扩大平台支持、SQL 语法覆盖和稳定性承诺。
- Lance 的 REST namespace 和企业 catalog 能否在权限、并发、治理上支撑团队级生产使用。
- 向量、全文、标量索引的维护成本，是否会在高频更新场景里保持可控。
- 与 Iceberg、Delta、Parquet 生态的互操作是否足够顺滑，避免形成新的数据孤岛。
- 真实业务里冷启动、随机 blob 读取、混合搜索的收益，是否能抵消引入新格式的学习成本。

## 结论

Lance 接入 DuckDB 的高阶信号是：AI 数据湖仓的竞争焦点正在从“谁扫描得更快”，转向“谁能把扫描、检索、更新、版本和多模态对象放在一个可治理的数据表里”。DuckDB 给它 SQL 入口，Lance 给它 AI 数据的底层形状。真正值得关注的不是某个 benchmark 的第一名，而是这套组合能否减少系统之间的数据搬运，让检索从外挂组件变成湖仓自己的能力。
