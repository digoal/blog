# 红队 · PG系防守派（产品战略分析师 + 生态护城河专家）

## 核心立场
**PG 系护城河是结构性优势，防守应为主旋律；贸然进攻反而会丢掉已有阵地。**

---

## 1. 复述与理解问题

非 PG 系来攻。调研报告已经标出了四个切入口：插件割裂场景、Oracle/Db2 深水区、低运维门槛市场、集分一体演进。

问题是：**PG 系该如何防？还有哪里能打？**

我的专业判断是：PG 系首先要把"已有阵地守住"，而不是分散资源四处出击。理由是：PG 系的优势是路线信任 + 生态惯性 + 人才密度，这三个护城河非 PG 系补起来需要 3-5 年，而 PG 系一旦放弃阵地，补回来要 5-10 年。

---

## 2. 核心论点

### 论点一：openGauss 35% 新增装机是结构性锁定，不是偶然
沙利文数据：2025 年线下集中式新增装机 openGauss 及 DBV 伙伴版 35.02%，位居第一。这不只是份额数字，背后是：
- DBV 伙伴体系：20+ 家商业发行版伙伴，每家都有本地化团队、本地招采资质、本地 DBA 培训体系
- 政府采购锁定：一旦项目用了 openGauss，三年维保、扩容、灾备、国测认证都在这个生态里
- 国密/安全可靠测评：已完成适配的厂商和版本，客户不会无故切换

这种"粘性锁定"是防守的根基。非 PG 系要抢走一个已上 openGauss 的政务客户，成本远高于新项目切入。

### 论点二：PG 语法生态是开发者的选择，不是厂商的选择
PostgreSQL 是全球最受开发者欢迎的数据库之一（StackOverflow 开发者调查连续多年前三）。国内大量云原生应用、互联网应用、AI 应用的第一选型是 PostgreSQL 或兼容 PostgreSQL 的产品。PG 系信创产品只需要维持"迁移成本最低"的承诺，就能承接这批客户进入信创场景的自然需求。

### 论点三：插件生态割裂是可以修复的，而且修复成本低于非 PG 系从头做
openGauss 社区已在推进插件框架；金仓、瀚高、Vastbase 在插件兼容性上均有投入。真正的挑战不是"技术上做不到"，而是"优先级排序"。PG 系厂商只需要对最高频的 5 个插件（PostGIS、pgvector、TimescaleDB、hstore、pg_partman）做到 90% 兼容，就能堵住非 PG 系的最大切入口。这是防守动作，不需要颠覆产品方向。

---

## 3. 第一性原理推导

前置条件：
1. 客户迁移数据库的核心驱动是"政策合规 + 风险可控 + 成本可预测"，不是"功能最强"
2. 已有数据库的替换成本 = 迁移人天 + 改写应用 + 测试周期 + 事故风险 + 人员培训

推导：
- 客户已用 openGauss/金仓/瀚高 → 替换成本极高 → 非 PG 系切入机会极小
- 客户新立项目 → PG 系靠语法兼容和开发者熟悉度仍有优势 → 防住新项目 = 防住未来盘子
- PG 系核心护城河 = 低切换成本 + 生态惯性 + 政策背书 → 三项中失去任何一项，护城河才会崩

结论：**PG 系防守重点不是"反击非 PG 系"，而是"不让客户有理由换"**

---

## 4. 适用边界

结论成立的条件：
- PG 系厂商能在 12 个月内补齐高频插件（PostGIS/pgvector）的 90% 兼容
- DBV 伙伴体系能在西部和下沉市场建立本地服务能力
- openGauss 社区能跟上 PostgreSQL 主版本节奏（不落后超过 2 个大版本）

结论失效的条件：
- PG 系多个头部厂商因商业竞争无法整合插件生态
- 非 PG 系厂商（如达梦、OceanBase）在金融核心拿到 10+ 标杆案例并公开宣发
- Oracle 解约或合规压力导致金融客户大规模快速迁移，PG 系来不及承接

## 6. 可证伪性

- **证明方法**：2026 年招采项目中，已用 openGauss/金仓的政务客户续签率 > 85%，则护城河成立
- **证伪方法**：若 2026 年出现 5+ 例政务/公安客户主动从 openGauss 切换到非 PG 系，护城河有裂缝
- **观测指标**：
  - openGauss DBV 伙伴年度装机数是否持续增长
  - 维保续约率和扩容比例
  - 招标文件中是否出现"去 openGauss"需求

**防守优先。PG 系护城河已经存在，核心任务是：补插件、建西部服务网络、跟上 PG 主版本。进攻方向只有一个：在金融分布式核心市场主动建标杆，不是为了全面进攻，而是为了证明 PG 系也能吃核心交易。**
