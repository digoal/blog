# PostgreSQL 系国产数据库：防守与进攻手册

> 写给产品规划决策者 | 2026-06-07

---

## 我要先戳破一个认知泡沫

很多 PG 系产品团队看到"线下集中式新增装机第一，35.02% 份额"这个数字，会不自觉地松一口气：我们已经是老大了，稳住就行。

这个判断是错的，而且是危险的错。

35% 的份额，大部分来自党政、安平、一般政务这些"最容易替换"的浅水区。根据市场调研，党政关键应用数据库国产替换率已约达 85%，这个盘子接近尾声。真正有大钱的战场——金融核心交易、工业时序物联网、AI 向量混合检索——PG 系并没有建立起结构性优势。非 PG 系国产数据库（达梦、OceanBase、TiDB、GoldenDB）拿着精准的切入口调研报告，正在往这几个方向集中资源。

所以我的核心判断是：**PG 系现在面临的，不是"如何稳守"的问题，而是"守什么、在哪里打"的战略资源分配问题。**

防守不能缺，但防守的重点不是守住已有份额，而是堵住真正致命的漏洞。进攻不能散，要集中力量打"非 PG 系还没站稳"的高价值市场。

---

## 防守手册：三道防线，一个都不能省

### 第一道防线：把插件生态从"最大漏洞"变成"护城河"

这是 PG 系当前最大的结构性弱点，必须第一优先级修复。

PostgreSQL Extension Network（PGXN）有 476 个扩展、3005 个发布版本。这是原生 PostgreSQL 最核心的竞争力之一——客户买的不只是一个 SQL 引擎，而是一套组合出来的业务能力。但国内 PG 系产品因为深度内核改造（openGauss 基于 PG 9.2 起步，历经多次国密、安全增强改造），与原生 PostgreSQL 插件 ABI 存在不同程度的割裂。

```
插件割裂风险矩阵（PG系视角）

插件类别        客户依赖深度    迁移壁垒强度    PG系当前风险
─────────────────────────────────────────────────────────
PostGIS         ★★★★★         ★★★★☆         高（空间函数覆盖不完整）
pgvector        ★★★★☆         ★★★★★         极高（AI应用快速渗透）
TimescaleDB     ★★★★☆         ★★★★☆         高（超表/连续聚合难替代）
HLL/bitmap      ★★★☆☆         ★★★☆☆         中（可做原生实现）
DuckDB FDW      ★★★☆☆         ★★★★☆         高（湖仓查询路径依赖）
pg_partman      ★★★☆☆         ★★★☆☆         中（分区管理可自研）
化学分子/图     ★★☆☆☆         ★★★★★         低（小众但粘性极强）
```

防守动作不是"追着每一个插件更新跑"，而是做三件事：

**第一件：发布插件兼容性公告。** 把每个 PG 系产品对高频插件的兼容程度、已知差异、迁移替代方案，整理成清单，主动告诉客户和集成商。这不是暴露弱点，而是建立信任——"我们知道差距在哪，我们告诉你怎么解决"比"我们全面兼容"这种没人信的说法强十倍。

**第二件：对 pgvector 和 PostGIS 做深度原生集成。** 这两个插件代表 AI 应用和空间应用两大高增长赛道。不需要做到覆盖所有函数，但必须做到核心路径（向量 ANN 检索 + 混合检索，PostGIS 空间索引 + 常用空间函数）在自己的内核里原生运行，而不是靠 shim 层适配。这是防御性产品投入，值得。

**第三件：建立"扩展场景迁移评估器"。** 这是一个小工具，输入客户当前 PostgreSQL 使用的扩展列表，输出：哪些能无缝迁移、哪些有差异、差异怎么解决。主动帮客户算清楚迁移成本，比客户自己踩坑再换供应商强多了。

---

### 第二道防线：西部和下沉市场的服务网络，必须提前布

当前 DBV 伙伴体系主要集中在华东、华北，这两个地区竞争已经非常充分。西部（西南、西北、东北）和下沉地区（地市、区县）的政务、医疗、教育、能源项目，本地 DBA 人才稀缺，客户真正在意的不是"哪个内核更正统"，而是"谁能交付、出事谁来救"。

如果 PG 系不在这里建立服务网络，这些市场会自然流向"服务能力更强的非 PG 系"。非 PG 系厂商已经看到了这个机会，调研报告明确提到：西部市场对"整包交付"的需求强于单点产品选型。

防守动作：在西部 5 个重点省（四川、陕西、云南、新疆、黑龙江）各发展 2-3 个有本地运维能力的 DBV 合作伙伴，提供标准化的"信创数据库整包"：License + 迁移评估 + 上线实施 + 运维监控 + 应急响应 + 信创适配证明文档。把西部服务能力从"空白"变成"有人守着"，进攻难度立刻对非 PG 系大幅提升。

---

### 第三道防线：主版本跟踪，不能继续落后

这是一个长期但致命的风险。openGauss 基于 PG 9.2 起步，国密改造和安全增强拉开了与上游的距离。PostgreSQL 本身仍在快速演进（16、17 已发布）。如果 PG 系产品在语法兼容性、扩展框架、优化器能力上持续落后于原生 PG，开发者就会倾向于在能直接用原生 PG 的场景下用原生 PG，信创场景才被迫用国产版——这会导致一个"信创专用、技术二流"的品牌形象，长期有害。

防守动作：在 openGauss 社区和 PG 系各发行版建立"主版本追踪路线图"，公开承诺每个主要版本的兼容计划和时间表。这是信任建设，而不只是工程任务。

## 进攻手册：三个市场，优先级排序明确

资源有限，不能三线同时打。优先级从高到低：

### 进攻第一优先：金融分布式核心交易

这是未来 3 年内最重要的市场机会，也是 PG 系当前最薄弱的地方。

IDC 数据：2024 年中国金融行业分布式事务型数据库市场 20.37 亿元，银行子市场 13.44 亿元；中小金融机构将在 2028 年前大量启动数据库选型。第一新声金融报告显示，超过三成金融机构核心系统国产数据库占比低于 20%，Oracle 仍是核心系统最主要的国外数据库。

这意味着什么？**大量金融机构还没选定国产数据库，选型窗口还开着。** 谁先拿到标杆案例、谁先建立"核心系统替换零事故"的口碑，谁就能拿到未来 5 年最大的钱。

```
金融核心数据库选型决策树（客户视角）

               ┌──────────────────────────┐
               │  需要替换核心交易数据库     │
               └────────────┬─────────────┘
                            │
              ┌─────────────▼──────────────┐
              │  分布式还是集中式？           │
              └──────┬──────────┬──────────┘
                     │          │
          ┌──────────▼──┐   ┌───▼──────────────┐
          │  分布式强一致 │   │  集中式高可用      │
          │  高并发多活   │   │  共享存储/主备     │
          └──────┬───────┘   └───┬──────────────┘
                 │               │
    ┌────────────▼──┐    ┌───────▼─────────────┐
    │ OceanBase/TiDB│    │ openGauss GaussDB   │
    │ GoldenDB/达梦 │     │ 金仓/瀚高/达梦集中式   │
    │【PG系弱区】    │     │【PG系有一定基础】      │
    └───────────────┘    └─────────────────────┘
```

PG 系进攻动作：

**第一步：选一到两家 PG 系厂商集中资源做金融核心分布式的技术突破。** 不是每家都打，而是联合 openGauss 社区，选一个最有资源的伙伴（如华为 GaussDB for openGauss 分布式版），集中工程力量攻克：两地三中心多活、强一致分布式事务、Oracle 存储过程高度兼容、无停机迁移、核心系统压测标准。

**第二步：用一个中型银行或证券公司的核心系统替换案例做"首发标杆"。** 案例要可公开、可验证、有第三方背书（审计机构或监管出具的安全可靠评估）。金融行业的客户决策极度依赖同业参考，一个真实的中型银行核心系统替换案例，价值胜过十个政务案例。

**第三步：把"Oracle 迁移到 PG 系"做成可复制的产品。** 这包括：Oracle 语法自动转换工具、存储过程分析报告、迁移风险评级、回退方案设计、迁移人天报价标准化。让迁移从"定制工程"变成"可预期产品"，客户的决策门槛大幅降低。

---

### 进攻第二优先：AI 向量 + 时序 + 空间的垂直场景

这三个场景的共同特点是：客户正在新建项目，还没有技术路线锁定；插件依赖重，但原生 PG 兼容库往往没有完整支持；迁移壁垒一旦建立极高。

```
三大垂直场景的进攻价值分析

场景        典型客户            进攻窗口      PG系动作
────────────────────────────────────────────────────────
向量检索    政务知识库、金融    2025-2027年    pgvector原生集成
            合规问答、安全情报  窗口开放中     + 混合检索支持
                                               
时序物联网  能源、工业、交通    2025-2028年    TimescaleDB替代
            基础设施监控        持续扩大       原生方案 + 压缩
                                               
空间GIS     自然资源、城规、    2024-2026年    PostGIS核心函数
            公安地图、农业      窗口快关闭     完整支持 + QGIS适配
```

进攻动作的核心是：**不要做"兼容 PG 扩展"这件事，而是做"原生集成、超越 PG 扩展"这件事。** 

举个具体例子：与其让客户在国产 PG 兼容库上装一个残缺版的 pgvector，不如在产品内核里原生实现向量索引（IVFFlat、HNSW）、支持向量 + BM25 混合检索、内置 embedding 模型调用接口。这样的产品，不只是"不比 pgvector 差"，而是"比在原生 PostgreSQL 上用 pgvector 更好用"。这才是真正的进攻，而不是追赶。

---

### 进攻第三优先：制造、能源、交通的"运维一体化"市场

这个市场的特点是：替换需求真实，但客户没有足够的 DBA，迁移风险是最大顾虑。第一新声报告显示，集中式数据库仍占 69%，说明大多数客户还没有分布式需求，但确实需要从 Oracle/Db2 迁出来。

PG 系的进攻武器不是技术，而是**确定性**。

进攻动作：发布"信创数据库替换 SLA 承诺书"，明确写清楚：
- 迁移评估报告出具时间（5 个工作日）
- 上线成功率承诺（≥99.9%）
- 事故响应时间（2 小时现场）
- 回退方案和回退时间（4 小时内）
- 第一年巡检和性能优化次数
- 漏洞修复最大响应周期

这不是免责声明，而是真正的产品竞争力。制造、能源、交通的甲方技术负责人签单前最怕的是"出了事没人管"，谁能把责任边界写清楚、让客户觉得"出事有人扛"，谁就赢。

## 如何验证这个判断是否跑偏

有五个信号我会持续跟踪：

**信号一**：2026-2027 年金融行业分布式数据库招标，PG 系和非 PG 系各自拿到多少中标。如果 PG 系一个都没有，说明进攻动作太慢；如果拿到 3 个以上，说明战略正确。

**信号二**：招标文件中是否出现"pgvector 兼容""PostGIS 完整支持"作为评分项。如果出现，说明插件生态的补强工作直接影响招标结果。

**信号三**：openGauss DBV 伙伴体系在西部省份是否有新注册的本地化公司。如果 6 个月内没有，西部市场会持续流失。

**信号四**：非 PG 系（OceanBase/达梦）是否在 2026 年发布超过 5 个金融核心系统替换案例。如果是，PG 系进攻金融核心必须提速。

**信号五**：政务 AI 知识库项目中，选型文件是否开始要求向量检索能力。如果是，AI 向量场景的进攻窗口已经打开。

## 最后说一句

PG 系最大的优势不是 35% 的份额，而是"PostgreSQL 语法是开发者的默认语言"这个事实。全球有几千万开发者写过 PostgreSQL，这个基础不会消失。PG 系要做的，是把这个语言优势转化为"核心系统替换的最低风险选项"。

防守插件生态，是为了让已有客户没有理由换；进攻金融核心，是为了让未来高价值客户的第一选项是 PG 系。这两件事不冲突，但优先级和资源分配必须想清楚。

浅水区的战役已经打完了。接下来的仗，在深水里。
