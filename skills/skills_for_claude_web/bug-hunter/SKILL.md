# Bug Hunter — 社会Bug猎手

## 核心理念

> **找这个世界的 "bug"（打眼一看就不合理的地方），想办法解决它，就能体现你的价值，同时可能有商机。**

本 Skill 的目标：从最新热点新闻中，用系统性视角找到那些"明明不对但大家习以为常"的现象，深度拆解根因，并给出可落地的解决路径。

---

## 执行步骤

### Step 1：获取热点新闻

**如果用户给出了行业名称**，搜索：
- `{行业名} 最新新闻 热点 2025`
- `{行业名} 问题 监管 争议`
- `{行业名} 事件 曝光`

**如果用户没有给出行业**，搜索社会热点：
- `中国 社会热点新闻 今日`
- `热点事件 争议 不合理`
- `消费 食品安全 医疗 教育 热点`

**搜索要求**：
- 至少搜索 3 轮，覆盖不同角度
- 优先选择近 30 天内、有争议性、涉及利益冲突的事件
- 选出 **2~3 个最具"bug感"的新闻事件**

**⚠️ 异常处理**：
- 如果搜索无结果 → 扩大关键词（如加"曝光""投诉""纠纷""骗局"），最多重试2轮
- 如果仍无结果 → 告知用户"该行业近期未发现典型Bug，建议更换行业或1个月后再试"
- 如果只有1个事件可用 → 降低要求至1个，但需补充1个跨行业类比案例

---

### Step 1.5：用户确认（Bug选择）

**展示备选Bug列表**，让用户确认或调整：

```
📋 已识别Bug候选（2~3个），请选择或补充：

[1] {事件A} — bug感：⭐⭐⭐⭐⭐
    直觉反应：{一句话描述哪里不对劲}

[2] {事件B} — bug感：⭐⭐⭐⭐
    直觉反应：{一句话描述}

[3] {事件C} — bug感：⭐⭐⭐
    直觉反应：{一句话描述}

请回复序号（如"1,2"或"1并补充电商行业"】，或直接描述您观察到的bug
```

**如果用户未回复**（等待超5分钟）→ 继续使用默认选择，但报告中注明"基于行业通用判断"

---

### Step 2：Bug识别与分析框架

对每个选出的事件，按以下维度展开分析，每项**不超过3句话**，防止空洞堆砌：

#### 🔍 Bug定位（表象）— 1句话
- 这件事打眼一看，哪里"不对劲"？
- 普通人的直觉反应是什么？

#### 🌐 根因拆解 — 每层1句，共3层
- **Why1**（为什么会这样）→ 直接原因
- **Why2**（为什么没人阻止）→ 结构性阻力
- **Why3**（为什么会一直持续）→ 激励/利益格局
- **利益链**（谁得利/谁受损）→ 用表格列出关键角色

#### 🏛️ 系统性分析 — 选2个框架深用
- **五力分析**：只写3行——权力最强方、动机最强方、维持现状的核心动力
- **信息不对称**：只写2行——谁有信息优势、谁处于盲区
- **监管盲区**：只写2行——失效的具体环节、执法成本为什么高
- **激励错位**：只写2行——每个参与者的错误激励是什么

#### 💡 解决思路 — 4类×3要素
每类解法必须包含：
1. **具体动作**（做什么，不是"加强监管"而是"强制实时上传数据"）
2. **执行路径**（谁来做，做什么检查）
3. **可行性评级**（⭐~⭐⭐⭐ + 理由1句话）
4. **需求侧真伪**（是真的痛点愿意付钱，还是伪痛点）

**供给侧/需求侧/痛点真伪格式**：
```
供给侧：能提供解决方案的主体（如技术公司/保险公司）
需求侧：真正愿意付费掏钱的人（如央国企IT部门/平台）
痛点真伪：✅真痛点 / ❌伪痛点（理由）
```

#### ⚠️ 二阶Bug（Bug里的Bug）— 1段话
- 写出最常见"解法"的局限性（1-2句话）
- 写出这个解法会让什么问题恶化

---

### Step 3：生成图形

**图形优先级（必须/可选）**：
1. ✅ Mermaid利益链条图（每个Bug必须）— 嵌入markdown，无需渲染
2. ✅ ASCII鱼骨图（综合视角必须）— 共性根因展示，对齐整洁
3. ✅ SVG四象限矩阵（综合视角必须）— 解决方案优先级矩阵

**SVG四象限矩阵规范**：
- viewBox="0 0 800 600"，深色背景#1a1a2e
- 四象限颜色：#2d6a4f(绿)/#9d4edd(紫)/#0077b6(蓝)/#d62828(红)
- 每个象限内放具体解决方案名称（8-12号字体）
- 坐标轴标注"影响力 →"和"可行性 →"
- **代码规范**：SVG代码直接书写，不使用反引号包裹；保留注释，注释单独成行；每个元素一行；代码内部不能有空行

---

### Step 4：输出格式

**输出路径**：`markdown/bug-hunter-{行业或关键词}-{YYYYMMDD}.md`
- 行业名为中文：直接拼音首字母（如"电商直播"→`bug-hunter-dszhibo-20260529`）
- 若markdown目录不存在 → 先创建：`/Users/digoal/new/markdown/`

## 决策树速查（TL;DR）

```
用户输入 → 有行业名？
  ├─ YES → 搜索3轮选Bug → Step1.5确认
  └─ NO → 搜索社会热点 → 选Bug → Step1.5确认

Step1.5 → 用户确认 → 分析2-3个Bug
  ├─ 每个Bug：表象→Why×3→利益链→系统分析→解决思路→二阶Bug
  ├─ 综合视角：共性根因→SVG矩阵→行动建议
  └─ 输出：markdown文件到markdown/

异常分支：
  ├─ 搜索0结果 → 扩大关键词重试2次 → 仍无则告知用户
  ├─ 只有1个Bug → 用1个+1个跨行业类比
  └─ 用户5分钟无回复 → 默认选择继续
```

## 检查点设计（防止自主失控）

| 检查点 | 位置 | 问题 | 处理 |
|--------|------|------|------|
| **Bug候选确认** | Step1.5 | 分析的Bug不是用户想要的 | 必须等用户回复后再深入分析 |
| **供给侧真伪** | Step2解决思路 | 解法无可行商业模式 | 每个解法必须写供给侧/需求侧/痛点真伪 |
| **图形完整性** | Step3 | 缺少必须图形 | SVG矩阵/ASCII鱼骨图/Mermaid缺一不可 |
| **输出路径** | Step4 | 目录不存在 | 先创建/Users/digoal/new/markdown/再写入 |

## 注意事项

- **保持批判性但建设性**：不是为了批评而批评，而是为了找到解法
- **数据支撑**：尽量引用具体数据、案例、政策文件
- **避免空洞结论**：每条解法都要有具体的执行路径，不要超过3句话
- **商机识别**：明确指出哪些bug的解法可能孕育商业机会
- **图形质量**：SVG要精美（viewBox规范、颜色对比清晰）、Mermaid逻辑清晰、ASCII对齐整洁
- **语言风格**：犀利、直接、有洞察力，不说废话，适合公众号/知识社群传播
