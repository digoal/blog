# Product Multi-Role Analysis Skill

## 总览

本 Skill 对任意产品进行「7 角色全景分析」，最终输出一份可直接发布或用于决策的综合报告。

**7 个分析角色：**
1. 👤 **用户** — 真实使用体验、痛点满足、愿意付费的理由
2. 💰 **投资人** — 市场规模、商业模式、增长曲线、退出路径
3. 🛠 **产品经理** — 核心功能、产品架构、roadmap 合理性、技术壁垒
4. 📣 **市场运营** — 获客渠道、增长策略、转化漏斗、ROI
5. 🎨 **品牌运营** — 品牌定位、视觉/语言一致性、用户心智占位
6. ⚔️ **友商** — 差异化威胁、可抄功能、对方弱点在哪
7. 🤝 **合作伙伴** — 合作切入点、互补价值、潜在风险

## 执行流程

### Phase 0 — 信息抓取与整理

**输入来源（按优先级）：**
- 用户提供的产品 URL → 使用 `web_fetch` 抓取官网/产品页
- 用户提供的文档（PDF/Word/Markdown）→ 读取内容
- 用户的文字描述 → 直接使用，并补充网络搜索

**必须收集的信息维度：**
```
产品名称、所属品类、目标用户、核心功能列表
定价策略、商业模式（SaaS/交易/广告/订阅等）
融资历史（轮次/金额/投资机构）
团队背景（创始人/核心成员）
主要竞品（至少3个）
用户评价（App Store/G2/Trustpilot/ProductHunt 等）
市场规模数据（TAM/SAM/SOM）
```

**搜索策略：**
- 搜索 `[产品名] review site:g2.com OR site:producthunt.com`
- 搜索 `[产品名] funding crunchbase`
- 搜索 `[产品名] vs [主要竞品]`
- 搜索 `[所属品类] market size [当前年份]`

将抓取结果整理为 `00_raw_intel.md` 保存到 `markdown/` 目录。

---

### Phase 1 — 7 角色独立分析（分文件保存）

每个角色的分析**独立保存**为单独的 Markdown 文件，命名规则：
```
markdown/01_user.md
markdown/02_investor.md
markdown/03_pm.md
markdown/04_growth.md
markdown/05_brand.md
markdown/06_competitor.md
markdown/07_partner.md
```

每个角色文件的**标准结构**，参见 `references/role-templates.md`。

**分析深度要求：**
- 每个角色视角至少 800 字
- 必须引用 Phase 0 中收集到的真实数据（不得凭空捏造数字）
- 每个角色有 1 个核心结论句（便于后续整合）
- 每个角色标注「该结论成立的前提假设」

---

### Phase 2 — 综合报告整合

将 7 个角色文件整合为最终报告 `markdown/[产品名]-analysis-report.md`。

报告结构详见 `references/report-structure.md`。

**图表生成规则：**
- SVG 图表 → 单独保存为 `markdown/assets/[图表名].svg`，在报告中用相对路径引用
- Mermaid 图表 → 直接嵌入报告的代码块
- ASCII 图表 → 直接嵌入，用于简单对比表格

**必须包含的图表（至少 4 张）：**
1. 产品竞争力雷达图（SVG）
2. 商业模式 / 收入结构图（Mermaid flowchart 或 SVG）
3. 用户旅程地图（Mermaid or ASCII）
4. 市场定位矩阵（SVG 四象限）

可选图表（根据产品类型选择）：
- 增长漏斗图
- 产品功能对比矩阵（竞品 vs 本品）
- 融资时间线

---

### Phase 3 — 结论边界与假设体系

这是本 Skill 区别于普通产品分析的核心价值。报告末尾**必须包含**：

#### 3.1 结论适用边界
```
本报告结论在以下条件下成立：
- 时间范围：[分析时间] ±12 个月内
- 地理范围：[主要市场，如：中国大陆 / 北美 / 全球]
- 竞争格局：[当前竞品格局未发生颠覆性变化]
- 宏观环境：[利率/监管/技术范式 未发生根本性转变]
```

#### 3.2 关键前提假设（KPA）
列出 3-5 个支撑核心结论的关键假设，格式：
```
KPA-1: [假设内容]
  观测指标: [用什么数据可以验证/证伪]
  验证周期: [多长时间可以看出端倪]
  
KPA-2: ...
```

#### 3.3 假设变动时的结论调整
用「如果…那么…」的条件句清晰表达：
```
如果 KPA-1 不成立（[具体情形]）→ 结论调整为：[修订结论]
如果 KPA-2 不成立（[具体情形]）→ 结论调整为：[修订结论]
```

## 注意事项

1. **数据真实性优先**：若找不到真实数据，明确标注「数据缺失，以行业均值估算」，禁止伪造权威引用。
2. **角色立场要鲜明**：每个角色视角要有真实的利益偏向，不能千篇一律说好话。友商视角必须写出威胁和攻击方向。
3. **图表要有信息量**：不要为图而图，每张图必须传递文字难以表达的洞察。
4. **结论要有勇气**：避免「可能」「也许」「或许」的模糊表达，结论要明确，不确定性用「假设体系」来承接。
5. **项目目录**：优先使用当前项目的 `markdown/` 目录；若不存在，创建 `/home/claude/markdown/` 并使用。
